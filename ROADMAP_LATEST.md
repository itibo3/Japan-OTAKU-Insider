## 1. 北極星（変えない軸）

**日本のオタクカルチャーのライフサイクル全体を、英語で追えるデータ基盤にする。**

- 点のニュース集約ではなく、作品ジャーニー（連載→書籍→漫画→アニメ→展開）を線で追えることを差別化軸にする
- 収集/翻訳/公開を分離し、公開サイトは軽量・静的運用を維持する
- 運用は「壊れない・止められる・別口がある」を優先する

## 3. 運用モード（課金/非課金の分岐）

## 3.1 非課金モード（普段）

- RSS収集: ローカル実行（`rss_fetch.py --fetch-thumbnails`）
- 翻訳: 手動（必要に応じてAI支援）
- 反映: `add_entry.py` で手動追加
- 公開: 手動コミット/プッシュ

## 3.2 課金モード（DeepL/検索API使用）

- 翻訳: DeepL APIで staging JSON を自動翻訳
- 収集補助: 検索APIを併用（将来統合）
- 公開: GitHub Actions日次実行（キー未設定時は安全スキップ）

## 3.3 将来モード（ローカルLLM）

- 翻訳API層をローカルLLMエンドポイントに差し替え
- 検索API層もローカル検索/エージェントに差し替え
- パイプライン構造は維持し、実装の置換コストを小さく保つ

## 8. 将来設計メモ（多言語・国内アフィ）

実装は後回し。方針だけ記録しておく。

- **管理1本化**: 完全別サイトにせず、リポジトリ1本で運用。URLは `/ja/` と `/en/` で分離し、別ページとして扱う
- **JSONは言語別ファイル**: `entries_ja.json` と `entries_en.json` に分けると後々楽（キャッシュ・差分更新・フロントロジックが単純）
- **日本語ページは日本語ソースをそのまま表示**: 表示時翻訳はしない。未翻訳（日本語マスタ）側のJSONを読むだけなので軽い
- **RSS Managerの書き出しを日英対応にしておくと便利**: パイプラインの入り口で言語別JSONを出せるようにしておくと、将来の自動化が楽
- **アナリティクス**: 別プロパティで分けても可。横断導線を見たい場合は1プロパティ＋言語ディメンション
- **SEO**: 日本語ページは `lang="ja"` と適切なURLで検索に載るようにする。`hreflang` で英語版と対応付け
